Lesson 0014

Pick the auth control from the exam clue.

Task Statement 2.1 is mostly service selection: IAM, STS, Cognito, API Gateway authorizers, Lambda policies, or least privilege.

The win

You will classify authentication and authorization scenarios before answer choices distract you.

Related review: domain triage and the DVA-C02 curriculum map.

The selection map

AWS code needs AWS APIs → IAM role Another account needs access → STS AssumeRole App users need sign-up/sign-in → Cognito user pool App users need AWS credentials → Cognito identity pool API callers sign with SigV4 → IAM authorization API needs bearer-token custom logic → TOKEN Lambda authorizer API needs headers/query/context → REQUEST Lambda authorizer Service must invoke Lambda → Lambda resource-based policy

1. IAM policy vs IAM role

Policy: the permission document. It grants or denies actions on resources.

Role: the assumable identity. It has a trust policy plus permissions. Choose roles for Lambda, ECS tasks, EC2, cross-account access, and temporary application credentials.

2. STS and AssumeRole

Choose AssumeRole when the clue says temporary credentials, delegated access, federation, or cross-account access. Avoid embedded long-term access keys unless the workload is truly outside AWS and no federation path is given.

3. Cognito vs IAM auth

User pools authenticate app users and issue tokens. Identity pools exchange identities for temporary AWS credentials mapped to IAM roles.

IAM auth means the API caller is an AWS principal that signs requests with Signature Version 4.

4. API Gateway authorizers

Use a Cognito/JWT authorizer for token validation managed by API Gateway. Use a Lambda authorizer when the scenario requires custom authorization logic.

TOKEN authorizer = bearer token. REQUEST authorizer = headers, query strings, stage variables, or context.

5. Lambda resource policies

Lambda calling out uses the execution role. Something invoking Lambda uses the function's resource-based policy. For least privilege, restrict principals, actions, resources, and source ARNs.

Fast distractor cuts

  • SCPs, permission boundaries, and session policies limit permissions; they do not grant access.
  • API keys are for usage plans and throttling, not real user authentication.
  • ECS task access uses task roles, not IAM groups or service-linked roles.
  • Cross-account access needs both trust and permissions.
  • Wildcard * is usually wrong when the question says most secure or least privilege.

Retrieval drill: choose the best answer

1. ECS tasks need access to AWS services.

2. A partner account needs temporary access.

3. Mobile users need sign-up and sign-in.

4. Signed-in users need temporary S3 credentials.

5. Internal callers sign API requests with SigV4.

6. Custom OAuth bearer-token logic is required.

7. Auth depends on headers and query strings.

8. API Gateway cannot invoke Lambda.

9. Lambda code must write to DynamoDB.

10. The question asks for least privilege.
