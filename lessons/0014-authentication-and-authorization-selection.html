Lesson 0014

Pick the auth control from the exam clue.

Task Statement 2.1 is service selection: IAM, STS, Cognito, API Gateway authorizers, Lambda policies, or least privilege.

Related review: domain triage and the curriculum map.

The selection map

  • AWS code needs APIs → IAM role | Another account needs access → STS AssumeRole
  • App users need sign-up/sign-in → Cognito user pool | App users need AWS credentials → Cognito identity pool
  • API callers sign SigV4 → IAM authorization | Bearer-token custom logic → TOKEN Lambda authorizer
  • API needs headers/query/context → REQUEST Lambda authorizer | Service invokes Lambda → Lambda resource policy

Key concepts at a glance

1. IAM policy vs role: policy grants/denies actions; role is an assumable identity with a trust policy. Choose roles for Lambda, ECS tasks, EC2, cross-account, and temporary credentials.

2. STS AssumeRole: temporary credentials for cross-account, federation, or delegated access. Avoid long-term keys unless there is no federation path.

3. Cognito: user pools authenticate app users and issue tokens; identity pools exchange identities for temporary AWS credentials mapped to IAM roles.

4. API Gateway authorizers: Cognito/JWT authorizer for managed token validation; Lambda TOKEN authorizer for bearer-token custom logic; Lambda REQUEST authorizer for headers, query strings, stage variables, or context.

5. Lambda policies: execution role controls what Lambda can call; resource-based policy controls who can invoke Lambda. Restrict principals, actions, resources, and source ARNs.

Fast distractor cuts

  • SCPs, permission boundaries, and session policies limit permissions—they do not grant access.
  • API keys are for usage plans and throttling, not user authentication.
  • ECS task access uses task roles, not IAM groups or service-linked roles.
  • Cross-account access needs both trust and permissions.
  • Wildcard * is wrong when the question says most secure or least privilege.

Retrieval drill: choose the best answer

1. ECS tasks need access to AWS services. Which IAM identity attaches to the task?

ECS task roles attach to tasks and provide temporary AWS credentials for service access.
IAM groups organize users—they do not attach to ECS tasks.
Root keys violate least privilege and should never be used in applications.
Access keys are long-term credentials—ECS tasks use temporary role credentials.
IAM role is correct. ECS task roles provide temporary credentials via STS.
Review IAM fundamentals and ECS task roles.

2. A partner AWS account needs temporary access to your resources. Which mechanism provides time-limited credentials?

AssumeRole returns temporary security credentials with optional expiration.
Static keys are long-term and violate least privilege for cross-account access.
Root login grants full account access—it is not a temporary access mechanism.
Shared users create security risks and violate least privilege.
STS AssumeRole is correct. It provides temporary credentials for cross-account or federated access.
Study cross-account access patterns and STS AssumeRole.

3. Mobile users need sign-up and sign-in for your app. Which Cognito feature authenticates users?

Cognito user pools authenticate users with sign-up, sign-in, and token issuance.
Identity pools exchange identities for AWS credentials—they do not handle sign-up/sign-in.
IAM roles are for AWS service access—not mobile user authentication.
API keys are for usage plans and throttling—not user authentication.
User pool is correct. User pools handle user authentication, sign-up, and token management.
Compare Cognito pools vs identity pools and Cognito user pools.

4. Signed-in users need temporary AWS credentials for S3 access. Which Cognito feature provides these?

Identity pools exchange Cognito or federated identities for temporary AWS credentials.
User pools issue tokens for authentication—they do not provide AWS credentials.
API keys are for usage plans—not temporary AWS credentials.
Stage caching stores responses—it does not provide AWS credentials.
Identity pool is correct. Identity pools exchange identities for temporary AWS credentials mapped to IAM roles.
Study Cognito identity pools and Cognito user pools.

5. Internal Lambda callers sign API requests with SigV4. Which authorization method validates signed requests?

IAM authorization validates Signature Version 4 signed requests from AWS principals.
API keys do not validate SigV4 signatures—they are for usage plans.
User pools authenticate app users with tokens—not SigV4 signatures.
Stage variables pass per-stage values—they do not validate signatures.
IAM authorization is correct. API Gateway validates SigV4 signatures from IAM principals.
Review API Gateway authorization and API Gateway IAM authorization.

6. Custom OAuth bearer-token logic is required for API Gateway. Which authorizer type fits this scenario?

TOKEN authorizers receive a bearer token and run custom validation logic in Lambda.
REQUEST authorizers receive the full request context—not a bearer token.
Usage plans control throttling and quotas—they do not handle authorization.
Stage caching stores responses—it does not handle authorization.
TOKEN Lambda authorizer is correct. It receives bearer tokens and runs custom authorization logic.
Review Lambda authorizer patterns and Lambda authorizers.

7. API Gateway authorization depends on request headers and query strings. The Lambda function needs full request context. Which authorizer type reads this context?

REQUEST authorizers receive headers, query strings, stage variables, and context.
TOKEN authorizers receive only the bearer token—not full request context.
Usage plans control throttling—they do not handle authorization.
Stage caching stores responses—it does not handle authorization.
REQUEST authorizer is correct. It receives headers, query strings, and full request context for Lambda-based authorization.
Study Lambda authorizer request context and Lambda authorizers.

8. API Gateway cannot invoke Lambda. Which Lambda policy grants invocation permission?

Lambda resource-based policies grant services permission to invoke the function.
Execution role controls what Lambda can call—not who can invoke Lambda.
User pools authenticate users—they do not grant Lambda invocation permission.
Access keys are long-term credentials—they do not grant invocation permission.
Resource policy is correct. Lambda resource-based policies grant API Gateway invocation permission.
Review Lambda resource-based policies and Lambda resource policies.

9. Lambda code must write to DynamoDB. Which IAM identity provides table access?

Lambda execution role grants the function permission to call AWS services like DynamoDB.
Resource policy controls who can invoke Lambda—not what Lambda can access.
Usage plans control throttling—they do not grant DynamoDB access.
Access tokens are for API authentication—not AWS service access.
Execution role is correct. Lambda execution role grants permission to write to DynamoDB.
Study Lambda execution roles and Lambda execution role.

10. A DVA-C02 scenario applies least privilege to an IAM policy. The policy must scope access to a specific resource. Which permission scope follows this principle?

Least privilege grants only the specific resource ARN needed—not broader access.
Wildcard * grants overly broad access and violates least privilege.
Root access grants full account permissions—it violates least privilege.
Shared keys create security risks and violate least privilege.
Specific ARN is correct. Least privilege scopes permissions to specific resources.
Review IAM least privilege patterns and IAM least privilege.
