Lesson 0009

Retry safely, then isolate poison failures.

DVA-C02 Domain 1 tests whether you can recognize safe retries, exponential backoff with jitter, idempotent logic, and dead-letter queues in distributed apps.

The win

When a scenario says transient failure, think retry. When it says duplicate side effects, think idempotency. When it says repeatedly failing message, think DLQ.

Related review: Lambda event source mappings and event-driven patterns.

The exam decision path

Is the failure temporary? yes → retry with exponential backoff + jitter no → fail fast or route to an error path Can the operation run twice safely? yes → retry normally no → add idempotency key / conditional write / processed-event record Does the same message keep failing? yes → send to DLQ after max receives / failed async attempts

1. AWS SDK retries are already built in

AWS SDKs include retry behavior for many transient errors, throttling responses, and network failures. Exam clue: if an app uses the AWS SDK and needs reliable API calls, prefer SDK retry configuration before custom retry loops.

Source: AWS SDK retry behavior.

2. Backoff waits longer; jitter spreads callers out

Exponential backoff increases delay after each failed attempt. Jitter randomizes that delay so thousands of clients do not retry at the same time and overload the service again.

Keyword trap: "retry immediately" can make an outage worse. "Reduce contention" or "avoid retry storms" points to jitter.

Source: AWS Architecture Blog: Exponential Backoff and Jitter.

3. Idempotency makes duplicate delivery harmless

Distributed systems can deliver duplicate events. Idempotent logic ensures the original request succeeds and later retries have no additional effect.

  • Use a client request token, order ID, or event ID as an idempotency key.
  • Record processed IDs in DynamoDB, often with a conditional write.
  • Design updates so repeating the same request does not double charge, double email, or double ship.

Source: AWS Lambda Powertools idempotency utility.

4. DLQs are for messages you cannot process now

A dead-letter queue captures failed messages after retry limits so normal processing can continue and operators can inspect, replay, or fix bad events.

  • For SQS, configure a redrive policy and maxReceiveCount.
  • For Lambda async invokes, use destinations or a DLQ for failed events.
  • Do not use a DLQ to hide bugs; alarm on DLQ depth.

Sources: SQS DLQs and Lambda async error handling.

Fast classification

  • ThrottlingException, 5xx, timeout: retry with SDK/backoff.
  • Many clients retry together: add jitter.
  • Payment/order/write may happen twice: make operation idempotent first.
  • One message blocks progress repeatedly: route to DLQ after max attempts.
  • Workflow step needs per-state retry/catch: Step Functions retry and catch policies.

Practice: choose the failure tool

1. A Lambda function uses the AWS SDK to write order records to DynamoDB during flash sales. The function occasionally receives ThrottlingException responses, and the team wants transient failures handled automatically with minimal custom code. What should the developer do?

Option A is correct because throttling is a transient failure, and AWS SDK retry modes already implement backoff-based retries for this pattern.
Option B is a distractor because alarms notify operators but do not automatically recover from short-lived throttling errors.
Option C is a distractor because a DLQ is for repeatedly failing messages, not for first-line handling of transient SDK API retries.
Option D is a distractor because a GSI is a DynamoDB index feature and has nothing to do with handling retry behavior in the client.
Option A is the correct answer. For transient DynamoDB throttling, the exam prefers the built-in AWS SDK retry strategy with exponential backoff over custom retry loops.

2. After a brief network outage, thousands of mobile clients reconnect and retry the same API calls at once. The development team wants to reduce the chance that synchronized retries will overwhelm the backend again. Which technique should the team add to the retry strategy?

Option A is correct because jitter randomizes retry timing so clients do not all retry in lockstep after the outage clears.
Option B is a distractor because more Lambda memory can improve function performance but does not spread out synchronized client retries.
Option C is a distractor because DAX is a read cache for DynamoDB and does not solve retry storm timing.
Option D is a distractor because alarms provide visibility but do not change how callers schedule retries.
Option A is the correct answer. Jitter is the key exam clue when the problem is many callers retrying at the same time.
Review Lesson 0009 on backoff and jitter. AWS SDK retry and backoff.

3. A checkout service calls an external payment processor. The processor might successfully charge the customer just before the service times out, so the client may retry the same request. The application must not create duplicate charges or duplicate orders. Which design best meets this requirement?

Option A is correct because an idempotency key plus a recorded processed state prevents the same logical request from creating duplicate side effects.
Option B is a distractor because reserved concurrency limits scaling; it does not make duplicate retries safe.
Option C is a distractor because visibility timeout applies to queue consumers, not to duplicate payment requests from retried API calls.
Option D is a distractor because CloudFront caching is unrelated to guaranteeing exactly-once business effects in a payment workflow.
Option A is the correct answer. The exam clue is duplicate side effects, which points to idempotency keys and conditional persistence.
Review Lesson 0009 on idempotency. AWS SDK retry behavior.

4. A Lambda function polls an Amazon SQS queue for invoice-processing jobs. One malformed message fails every time, but the rest of the queue should continue processing and operators need to inspect the bad payload after a set number of attempts. Which configuration should the developer use?

Option A is correct because an SQS redrive policy moves messages that exceed maxReceiveCount into a dead-letter queue for later inspection.
Option B is a distractor because DynamoDB TTL expires table items, not SQS messages that repeatedly fail processing.
Option C is a distractor because EventBridge archiving is unrelated to poison-message handling for SQS consumers.
Option D is a distractor because Lambda layers share code packages but do not change failure routing for queue messages.
Option A is the correct answer. A dead-letter queue is the standard AWS pattern for isolating poison messages after repeated failures.
Review Lesson 0008 on event-driven patterns. Amazon SQS DLQ.

5. An order-fulfillment workflow is implemented as a state machine. The payment task must retry only on throttling errors, while the inventory task must catch a business exception and branch to a compensation path. Which AWS service feature supports this per-step error handling?

Option A is correct because Step Functions lets each state define its own Retry and Catch behavior.
Option B is a distractor because API Gateway gateway responses customize HTTP error responses, not workflow state error handling.
Option C is a distractor because SNS delivery retries apply to message delivery attempts, not to branching logic inside a workflow state machine.
Option D is a distractor because CloudWatch alarms observe system behavior but do not execute retries or catches within a workflow.
Option A is the correct answer. Per-state retry intervals and catch handlers are a native Step Functions capability.
