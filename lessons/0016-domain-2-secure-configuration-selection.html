Lesson 0016

Choose the secure config store before wiring Lambda.

DVA-C02 Domain 2 secure-configuration questions usually test one fork: secret, parameter, KMS key, environment variable, or certificate.

The win

When the scenario says credentials, connection strings, encryption, rotation, or environment variables, classify the data before choosing the service.

Related review: Lambda configuration traps and security error handling.

The exam decision path

Database password / API key / rotation required? → AWS Secrets Manager Hierarchical app config or simple encrypted string? → SSM Parameter Store SecureString + KMS Per-function non-secret setting? → Lambda environment variable Encrypt data locally or use envelope encryption? → AWS KMS GenerateDataKey TLS certificate for a website or load balancer? → AWS Certificate Manager Direct control of dedicated HSM hardware? → AWS CloudHSM

1. KMS manages keys, not application secrets

AWS KMS creates and controls cryptographic keys. Use it for server-side encryption integrations and envelope encryption. For local encryption, call GenerateDataKey, encrypt with the plaintext data key, erase it from memory, and store the encrypted data key with the ciphertext.

Source: AWS KMS Developer Guide.

2. Secrets Manager vs Parameter Store

  • Secrets Manager: credentials, API keys, RDS passwords, built-in rotation, KMS encryption, resource policies.
  • Parameter Store: hierarchy paths, config values, versioning, labels, plaintext strings, or encrypted SecureString values.
  • Exam shortcut: if rotation is required, choose Secrets Manager.

Sources: Secrets Manager and Parameter Store.

3. Lambda environment variables are not shared secret storage

Lambda encrypts environment variables at rest. A customer managed KMS key can add control, but environment variables are per function and can be visible to principals allowed to view function configuration. For shared sensitive values, retrieve Secrets Manager or Parameter Store values at runtime, often with a Lambda extension or cache.

Source: Lambda environment variable encryption.

4. Configuration pitfalls the exam likes

  • Stage variable: API Gateway stage-specific value passed to an integration.
  • Environment variable: function-specific setting, not cross-function sharing.
  • SecureString: encrypted Parameter Store value using KMS.
  • Secrets extension: retrieves and caches secrets without writing SDK boilerplate.
  • Interface endpoint: private VPC access to Secrets Manager or SSM APIs.

Exam traps

  • KMS is not where you store the password. KMS protects keys used by services that store data.
  • Parameter Store can be plaintext. Look for SecureString when encryption is required.
  • Lambda env vars are per function. They do not share one secret across functions.
  • CloudHSM is a distractor. Choose it only when the scenario requires dedicated HSM control.
  • ACM is for TLS certificates. It is not for database password rotation.

Practice: choose the secure configuration

1. A company runs an application on AWS Lambda that connects to an Amazon RDS for MySQL database. Security policy requires the database credentials to be encrypted at rest and rotated automatically every 30 days with minimal custom code. Which AWS service should the developer use?

Correct. Secrets Manager is purpose-built for secrets such as database credentials and supports native rotation.
Plaintext Parameter Store does not satisfy the rotation requirement and is not the secure answer for database credentials.
ACM manages TLS certificates, not application database passwords.
Lambda environment variables can hold values, but they do not provide built-in automatic credential rotation.
Option A is the correct answer. Encrypted database credentials with automatic rotation is the textbook AWS Secrets Manager scenario.
Study Lesson 0015 and Secrets Manager.

2. Five Lambda functions in the same application need the same encrypted database connection string. The value changes rarely, and the team wants to store it once and retrieve it at runtime rather than duplicating it in every function configuration. Which approach is the best fit?

Correct. Parameter Store SecureString keeps one encrypted value that multiple functions can read securely.
Per-function environment variables duplicate the value and make coordinated updates harder across functions.
Stage variables are for API Gateway integration settings, not shared secret storage for Lambdas.
A CloudFront cache key defines caching behavior and has nothing to do with configuration storage.
Option A is the correct answer. A shared encrypted configuration value with rare updates is a strong fit for Parameter Store SecureString.
Review Lesson 0015 and Parameter Store.

3. A Lambda function reads from a DynamoDB table whose name differs by deployment environment. The value is not sensitive, and the developer wants a simple per-function setting available at runtime. Where should this configuration be stored?

Correct. Environment variables are the normal place for non-secret per-function settings such as table names.
Secrets Manager is better for credentials and sensitive values, not simple non-secret configuration.
ACM stores certificates and does not hold Lambda runtime settings.
CloudHSM provides dedicated hardware key management and is unrelated to plain configuration values.
Option A is the correct answer. A non-secret value that belongs to one Lambda function should live in a Lambda environment variable.
Study Lesson 0005 and Lambda environment variable encryption.

4. A desktop application must encrypt large files locally before uploading them to Amazon S3. The company wants to use AWS KMS but avoid sending the full file contents to KMS for every encryption operation. Which KMS feature should the developer use?

Correct. GenerateDataKey provides a plaintext data key for local encryption plus an encrypted copy to store with the ciphertext.
Direct KMS Encrypt on large file contents is not the efficient pattern for bulk data encryption.
ACM manages certificates and does not generate data keys for envelope encryption.
S3 server-side encryption protects objects after upload, not local encryption before upload.
Option A is the correct answer. Envelope encryption with GenerateDataKey keeps KMS focused on protecting the small data key instead of the whole file.
Review Lesson 0005 and KMS envelope encryption.

5. A Lambda function starts failing to decrypt data after the team switches from the AWS managed KMS key to a customer managed key. The execution role already has IAM permissions for KMS actions. What should the developer check next?

Correct. Customer managed KMS keys also require the key policy or grant to allow the caller.
A DLQ captures failed events but does not control whether KMS allows decryption.
Memory size does not affect KMS authorization decisions.
S3 lifecycle settings do not determine whether a principal can use a KMS key.
Option A is the correct answer. KMS access often requires both IAM permission and a key policy that allows the Lambda principal.
Review Lesson 0015 and KMS key policies.

6. A workload runs in private subnets and must call AWS Secrets Manager without using an internet gateway or NAT device. Which VPC feature should the developer add so the application can reach the Secrets Manager API privately?

Correct. An interface endpoint exposes the service privately inside the VPC through AWS PrivateLink.
A NAT gateway still sends traffic toward the public internet path and does not meet the no-internet requirement.
A Lambda function URL is a public invoke feature and does not provide private access to Secrets Manager.
An S3 gateway endpoint is only for S3, not for Secrets Manager APIs.
Option A is the correct answer. Private subnet access to Secrets Manager without internet routing requires an interface VPC endpoint.
Study Lesson 0017 and VPC endpoints.
