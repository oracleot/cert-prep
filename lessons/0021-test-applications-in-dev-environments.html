Lesson 0021

Test deployments without touching production.

Task Statement 3.2: pick the safest dev/test pattern when the scenario mentions versions, mock APIs, stages, templates, or EC2 bootstrapping.

The win

You will classify Domain 3 test-environment questions by the clue: deployment risk, Lambda promotion, API dependency, SAM template, or CloudFormation helper script.

Source review: Domain 3 Task 3.2 in docs/aws dva-c02.md.

Fast selector

Least time, highest blast radius → all-at-once Major runtime/platform change → blue/green Stop app, install version, validate → in-place EC2 package/files/services from metadata → cfn-init Wait for EC2 config before stack continues → cfn-signal Deploy one SAM template, not project root → sam deploy --template Backend not ready, API consumers must test → API Gateway mock integration Same API, different dev/test config → API Gateway stages + variables Stable event source while Lambda changes → Lambda alias ARN

1. Choose the deployment strategy

  • All-at-once is fastest, but the riskiest. Pick it when speed matters and downtime/blast radius is acceptable.
  • Blue/green is the exam answer for major runtime, server, or platform changes because the new environment is separate.
  • In-place means stop the app, install the latest version, start it, then validate it on the same resources.
  • Combine tools when needed: CloudFormation manages infrastructure; CodeDeploy, Elastic Beanstalk, ECS, or similar services manage application updates.

2. Promote Lambda safely

Exam shortcut

Versions are immutable. Aliases are movable pointers. Point S3, API Gateway, or other event sources at the alias ARN so promotion only changes the alias target.

  • Develop against $LATEST, publish a numbered version, then test that version in dev or beta.
  • Move aliases such as dev, beta, and prod to approved versions as testing completes.
  • Do not make every event source update its function ARN after each release; use the alias ARN as the stable integration target.

3. Mock and stage APIs

  • Mock integration lets API Gateway return configured responses when the backend service is missing, unstable, or not ready for consumer tests.
  • Stages represent deployed API environments such as dev, test, beta, and prod.
  • Stage settings can enable caching, throttling, logging, and canary testing per environment.
  • Stage variables hold environment-specific values, such as which backend or Lambda alias a stage should call.

4. Template and EC2 bootstrap clues

  • Use sam deploy --template when the question asks SAM to deploy a specific template instead of the current project root.
  • Use cfn-init to read CloudFormation metadata and install packages, create files, enable services, or start services on EC2.
  • Use cfn-signal with a CreationPolicy or wait condition when CloudFormation must wait for an instance to finish configuration before continuing.
  • Prebaked AMIs reduce instance startup work; bootstrapping configures instances at launch.

Mixed drill: choose the best answer

1. A test API needs responses before the backend exists.

2. Same API needs dev and prod configuration values.

3. S3 should invoke prod Lambda without ARN updates.

4. A major platform version changes with lower cutover risk.

5. EC2 must install packages from template metadata.

6. CloudFormation must wait until EC2 finishes setup.

7. SAM must deploy a specific staging template.

8. The fastest deployment can tolerate full blast radius.
