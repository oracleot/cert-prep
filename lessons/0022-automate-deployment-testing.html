Lesson 0022

Automate the test gate before production.

Task Statement 3.3: choose the AWS service or feature that makes deployment testing repeatable across environments.

The win

You will classify deployment-testing clues into pipeline stages, buildspec tests, traffic shifting, API integrations, and approved environment promotion.

Source map: Domain 3 Task 3.3 in docs/aws dva-c02.md.

Exam selector

Commit triggers release workflow → CodePipeline source stage Run unit tests from commands → CodeBuild buildspec Shift Lambda traffic every interval → CodeDeploy linear Shift a small slice, then the rest → CodeDeploy canary Route all traffic immediately → CodeDeploy all-at-once Promote only approved container image → ECR tags or digests

Buildspec clue

If the scenario says install dependencies, run unit tests, package artifacts, or publish test reports from commands, choose CodeBuild buildspec.yml.

Retrieval drill: pick the tested feature

1. A DVA-C02 scenario describes a commit to a Git repository that must trigger build, test, staging, and production actions automatically. You must name the orchestration service. Which AWS service chains source, build, test, and deploy stages triggered by code changes?

CodePipeline orchestrates ordered stages (source, build, test, deploy) triggered by repository changes—Task 3.3.
CodeArtifact stores package dependencies; it does not orchestrate pipeline stages.
CodeCommit hosts the Git repository; it does not chain pipeline stages.
CodeGuru performs code reviews; it does not orchestrate CI/CD stages.
CodePipeline is correct. It is the release automation spine that chains stages on commit events.
Review Task 3.4 CI/CD services and CodePipeline.

2. A scenario involves a CodeBuild project where unit tests and lint commands must run during the build phase. You must name the configuration file. Which YAML file defines install, build, and post-build commands that CodeBuild executes?

buildspec.yml defines install, pre_build, build, and post_build phases for CodeBuild—Task 3.3.
AppSpec file configures CodeDeploy application and deployment lifecycle hooks; it does not run in CodeBuild.
CloudFormation template defines infrastructure resources; it does not run build commands.
Runbook documents operational procedures; it is not a CodeBuild configuration file.
buildspec.yml is correct. CodeBuild reads this file to run the install, build, and test phases of the project.
Study Task 3.4 CI/CD services and CodeBuild buildspec reference.

3. A DVA-C02 scenario describes a CodeDeploy Lambda deployment where traffic shifts to the new version in equal increments at fixed intervals. You must name the traffic configuration. Which CodeDeploy Lambda deployment configuration shifts traffic in equal percentages at fixed intervals?

Linear shifts equal traffic percentages at fixed intervals (e.g., 10 percent every 10 minutes)—Task 3.3.
Canary shifts an initial small percentage, then shifts the remainder at once—fixed equal increments are not canary.
Blue/green deploys to a separate environment before shifting traffic; it is not equal increments.
Immutable deploys to a new fleet and terminates the old one; it does not shift traffic incrementally.
Linear is correct. CodeDeploy linear configuration shifts traffic in equal increments at fixed intervals.
Review CodeDeploy configurations and Task 3.4.

4. A scenario involves an API Gateway integration that must return test responses before any Lambda backend exists. You must choose the correct integration type. Which API Gateway integration type returns a configured response without invoking any backend?

MOCK integration returns fixed responses configured in API Gateway without calling any backend—Task 3.3.
HTTP integration calls an HTTP backend; it requires a live endpoint.
AWS integration invokes an AWS service or Lambda; it requires a real backend.
VPC is not an API Gateway integration type.
MOCK is correct. It enables API Gateway to return configured responses for contract testing or when backends are unavailable.
Study API Gateway integration types and Task 3.2.

5. A DVA-C02 scenario involves a pipeline that must deploy to another AWS account using temporary, time-limited credentials. You must name the security mechanism. Which AWS service provides temporary, role-based credentials for cross-account pipeline access?

STS AssumeRole returns temporary security credentials with an expiration time for cross-account access—Task 3.3.
KMS keys encrypt and decrypt data; they do not provide temporary credentials.
SCPs (Service Control Policies) restrict permissions in AWS Organizations; they do not issue credentials.
Network ACLs filter subnet traffic; they do not provide cross-account credentials.
STS AssumeRole is correct. It is the standard mechanism for pipelines to assume cross-account IAM roles with temporary credentials.
Review Task 3.4 CI/CD and IAM AssumeRole.

6. A DVA-C02 scenario involves a frontend application where test and production environments must follow Git branches automatically. You must name the service. Which AWS service connects Git branches to frontend and backend environments, deploying updates on every commit?

Amplify connects Git branches to hosting environments and deploys automatically on commits—Task 3.3.
AppConfig manages runtime feature flags and configurations; it does not connect Git branches to environments.
OpsWorks manages configuration with Chef or Puppet; it does not follow Git branch deployments.
DataSync transfers data between storage systems; it does not manage deployments.
Amplify is correct. It maps Git branches to environments and triggers automatic deployments on code commits.
Study AWS Amplify and Task 3.4 CI/CD.
