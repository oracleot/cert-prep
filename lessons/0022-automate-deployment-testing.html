Lesson 0022

Automate the test gate before production.

Task Statement 3.3: choose the AWS service or feature that makes deployment testing repeatable across environments.

The win

You will classify deployment-testing clues into pipeline stages, buildspec tests, traffic shifting, API integrations, and approved environment promotion.

Source map: Domain 3 Task 3.3 in docs/aws dva-c02.md.

Exam selector

Commit triggers release workflow → CodePipeline source stage Run unit tests from commands → CodeBuild buildspec Shift Lambda traffic every interval → CodeDeploy linear Shift a small slice, then the rest → CodeDeploy canary Route all traffic immediately → CodeDeploy all-at-once Promote only approved container image → ECR tags or digests

1. CodePipeline is the automation spine

  • Stages group actions in order: source, build, test, staging, approval, production.
  • Source starts from a repository or artifact change; build/test often runs CodeBuild.
  • Deploy can call CloudFormation, CodeDeploy, ECS, Elastic Beanstalk, or custom actions.
  • Cross-account deployments use IAM roles and STS AssumeRole so the pipeline can enter test, pre-prod, or prod accounts with temporary credentials.

2. CodeBuild proves the artifact

Buildspec clue

If the scenario says install dependencies, run unit tests, package artifacts, or publish test reports from commands, choose CodeBuild buildspec.yml.

install dependency setup pre_build login, lint, test setup build compile and unit tests post_build package, report, push artifact reports test result groups artifacts files for the next stage

3. CodeDeploy controls blast radius

  • Canary: send a small percentage first, wait, then send the rest. Clues: 10% first, monitor, then full.
  • Linear: shift equal increments at fixed intervals. Clues: every 5 minutes, every 15 minutes, equal batches.
  • All-at-once: shift all traffic immediately. Fastest, riskiest, least validation window.
  • Validation: use lifecycle hooks and CloudWatch alarms to stop or roll back bad deployments.

4. API Gateway testing keywords

MOCK Gateway returns test responses without a backend AWS_PROXY Lambda proxy; full request event passed to Lambda AWS AWS service or Lambda with mapping templates HTTP_PROXY Passthrough to HTTP backend HTTP HTTP backend with request/response mapping
  • Use stages for dev, test, staging, and prod API deployments.
  • Use stage variables, logging, throttling, caching, and canary settings to isolate test behavior.
  • Choose MOCK when dependencies are unavailable but consumers need predictable integration tests.

5. Approved versions across services

  • ECR tags mark images such as dev, test, release, or prod; immutable digests are safer for exact approvals.
  • Amplify branches map Git branches to frontend/backend environments and deploy updates on commits.
  • AWS Copilot creates ECS application environments and can provision pipelines across accounts or Regions.
  • STS roles let automated tests assume only the account role needed for that environment.

Retrieval drill: pick the tested feature

1. A commit must trigger build, test, staging, and production actions.

2. Unit tests must run from commands during the build.

3. Lambda traffic shifts to the new version every 15 minutes.

4. API consumers need test responses before the backend exists.

5. API Gateway must pass the full request event to Lambda.

6. Only an approved container build should enter integration tests.

7. Frontend test and prod environments follow Git branches.

8. A pipeline deploys tests into another account with temporary credentials.
