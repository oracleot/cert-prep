Lesson 0010

Do DynamoDB exam math before choosing features.

DVA-C02 Domain 1 tests whether you can calculate capacity, spot consistency clues, and avoid index and partition traps.

The win

Given a DynamoDB scenario, classify it in this order: access pattern → capacity math → consistency → index behavior → partition risk → cache fit.

Related review: Domain 1 triage.

1. Capacity math: round up first

  • 1 WCU = one write per second for an item up to 1 KB. Round item size up to the next KB.
  • 1 RCU = one strongly consistent read per second for an item up to 4 KB. Round item size up to the next 4 KB.
  • Eventually consistent reads cost half: one RCU gives two eventual reads for up to 4 KB items.
  • Transactional reads/writes cost double; do not pick them unless the scenario needs all-or-nothing correctness.
WCU = writes/sec × ceil(item KB / 1) Strong RCU = reads/sec × ceil(item KB / 4) Eventual RCU = (reads/sec × ceil(item KB / 4)) / 2

2. Consistency: base table yes, GSI no

DynamoDB tables and local secondary indexes can use strongly consistent reads. Global secondary indexes support eventual consistency only.

Exam clue: if the answer requires strong reads on an alternate key, a GSI is a trap.

3. Query beats Scan

  • Query: targeted. Requires a partition key value and can narrow with a sort key condition.
  • Scan: reads every item or index entry, then filters. Usually expensive and slow.
  • FilterExpression: reduces returned items, not capacity consumed. It is not a capacity-saving strategy.

4. GSI behavior to recognize

  • A GSI has its own partition/sort key and lets you query by a different access pattern.
  • GSI reads and writes consume capacity from the index, not the base table.
  • Base-table writes can be throttled if the GSI cannot keep up.
  • Projected attributes decide what the index can return without fetching from the table.

5. Hot partitions are key-design failures

If consumed table capacity is below provisioned capacity but requests are throttled, suspect a hot partition: one partition key gets too much traffic.

  • Use high-cardinality partition keys: many distinct values, evenly accessed.
  • Use composite keys when the access pattern needs grouping plus ordering.
  • For write-heavy skew, add random suffix sharding and query the shards when reading.
  • Do not solve a hot write partition with DAX; DAX is a read-through cache.

6. DAX: read cache, not correctness

Choose DynamoDB Accelerator (DAX) for microsecond response times on read-heavy, eventually consistent workloads. Avoid it when the scenario needs strongly consistent reads, write scaling, ad hoc analytics, or better partition-key distribution.

Practice: exam triage

1. A gaming company stores 4 KB player profile items in a DynamoDB table provisioned with 10 read capacity units. The application can use eventually consistent reads for these lookups. What is the maximum number of reads per second the table can support for this workload?

Option A is a distractor because 5 reads per second would undercount the capacity available from 10 RCUs.
Option B is a distractor because 10 RCUs support 10 strongly consistent 4 KB reads per second, but eventual consistency doubles that throughput.
Option C is correct because each RCU supports two eventually consistent 4 KB reads, so 10 RCUs support 20 reads per second.
Option D is a distractor because 40 reads per second would require twice as many RCUs for 4 KB items.
Option C is the correct answer. Eventually consistent reads cost half an RCU for 4 KB items, so 10 RCUs provide 20 reads per second.

2. A fitness application stores workouts with userId as the partition key and workoutId as the sort key. The app must also query each user's workouts by workoutDate, and those alternate-key reads must support strong consistency. Which DynamoDB index type should the developer add?

Option A is a distractor because GSIs support eventual consistency only, which does not meet the strong-read requirement.
Option B is a distractor because DAX is an eventually consistent cache and does not create an alternate sort key for querying.
Option C is correct because an LSI keeps the same partition key, adds an alternate sort key, and supports strongly consistent reads.
Option D is a distractor because scanning with a filter is inefficient and still does not provide the indexed strong-read access pattern the scenario needs.
Option C is the correct answer. The strong-consistency clue eliminates GSI and points to an LSI on the same partition key.
Review Lesson 0010 on DynamoDB indexes. Amazon DynamoDB indexes.

3. A multi-tenant SaaS platform stores events in DynamoDB and has enough total write capacity provisioned. However, one large tenant still gets frequent throttling because most writes use the same partition key value. Which design change best distributes the write traffic?

Option A is a distractor because DAX accelerates reads and does not spread write traffic across partitions.
Option B is a distractor because scans increase read work and do nothing to solve a write hot-partition problem.
Option C is correct because partition-key sharding spreads writes across multiple physical partitions instead of concentrating them on one hot key.
Option D is a distractor because a GSI with the same partition key does not change the skew on the base table's hot key.
Option C is the correct answer. When overall capacity is sufficient but a single key is hot, the fix is to redesign the partition key distribution.
Review Lesson 0010 on hot partitions. DynamoDB partition key design.

4. An advertising platform repeatedly reads the same product metadata from DynamoDB during each page render. The workload is read-heavy, eventual consistency is acceptable, and the team needs microsecond latency without redesigning the table schema. Which DynamoDB feature is the best fit?

Option A is correct because DAX is an in-memory cache that provides microsecond latency for read-heavy, eventually consistent DynamoDB access.
Option B is a distractor because an LSI changes query options but does not cache hot reads in memory.
Option C is a distractor because Streams capture item changes for downstream processing, not low-latency read acceleration.
Option D is a distractor because transactional writes increase correctness guarantees for writes and are unrelated to read caching.
Option A is the correct answer. DAX is the exam signal for microsecond, eventually consistent DynamoDB reads.
