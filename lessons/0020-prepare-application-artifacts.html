Lesson 0020

Prepare the artifact before you deploy.

Task Statement 3.1: choose the build output, package command, repository, or configuration source the scenario is really testing.

The win

You will separate dependency artifacts, serverless packages, container images, and runtime configuration without over-selecting CI/CD services.

Source map: Domain 3 Task 3.1 in docs/aws dva-c02.md.

Artifact selector

Package dependencies for teams → AWS CodeArtifact Store SAM/Lambda zip artifacts → Amazon S3 Build Lambda-native dependencies → sam build --use-container Zip and upload SAM artifacts → sam package Create or update the stack → sam deploy Declare SAM inside CloudFormation → Transform Store container deployment images → Amazon ECR Point traffic without changing ARNs → Lambda alias

1. Repository vs package vs deploy

  • CodeArtifact is a managed artifact repository for package managers such as npm, pip, Maven, and NuGet.
  • S3 is the common storage target for packaged SAM and Lambda zip artifacts before CloudFormation deploys them.
  • sam build prepares the application locally. Use --use-container when native dependencies must match the Lambda runtime.
  • sam package zips local artifacts, uploads them to S3, and writes a packaged template for deployment.
  • sam deploy uses the packaged template to create or update the CloudFormation stack.

2. CloudFormation and Lambda clues

Exam shortcut

If the template includes SAM resources, look for Transform: AWS::Serverless-2016-10-31. If the question says deploy code plus dependencies, decide between zip, layer, or container image.

  • Zip package: upload directly if small enough, or put the package in S3 for Lambda/CloudFormation to use.
  • Lambda layer: share libraries, custom runtimes, or stable dependencies across zip-based functions.
  • Container image: package runtime, code, and dependencies together; Lambda layers are not used with container images.
  • Lambda alias: stable pointer to a published version, useful when event sources should keep the same ARN during promotion.

3. Containers and configuration

  • ECR stores Docker/container images used by ECS tasks and Lambda container-image functions.
  • ECS task definitions reference the image URI and define container settings such as port mappings and environment variables.
  • AppConfig manages and validates runtime application configuration for EC2, Lambda, containers, mobile, and IoT apps.
  • cfn-init applies CloudFormation metadata on EC2 to install packages, write files, and start services during provisioning.

Retrieval drill: pick the deployment clue

1. Teams need a managed npm artifact repository.

2. SAM must zip code, upload S3, write template.

3. Native Lambda dependencies must match Amazon Linux.

4. SAM resources appear inside a CloudFormation template.

5. Lambda zip is too large because dependencies repeat.

6. ECS tasks need an immutable Docker image source.

7. EC2 stack should install packages from template metadata.

8. Runtime feature flags need validation and safe rollout.
