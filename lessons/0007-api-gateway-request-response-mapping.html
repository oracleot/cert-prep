Lesson 0007

API Gateway mapping is where requests and responses change shape.

DVA-C02 tests whether you can place validation, parameter mapping, templates, and status-code handling on the correct side of API Gateway.

The win

When a scenario says "transform," "rename," "require," or "return a different status," first classify the flow: method request → integration request → backend → integration response → method response.

Anchor this to Domain 1 development: API Gateway is being tested as the app integration layer.

The five boxes

Client → Method request: public API contract, auth, required params, validation → Integration request: transform client input for Lambda, HTTP, or AWS service → Backend → Integration response: map backend output, headers, errors, status overrides → Method response: public response contract: status codes, models, headers

Method request: reject bad input early

Use method request settings to declare request parameters, such as a required query string. Add a request validator when API Gateway must check required parameters or request bodies before calling the backend.

Integration request: reshape the call

Use integration request mapping when the backend expects a different shape than the public API. In REST APIs, non-proxy integrations can use Velocity Template Language (VTL) mapping templates to build a Lambda event or an AWS service request.

Integration response: reshape the answer

Use integration responses to transform backend output into the public API response. This is where error patterns, headers, body templates, and $context.responseOverride.status can change what the client receives.

Method response: declare the contract

Method responses define which status codes, response headers, and models the API can return. If a header or status is not declared for a REST API method response, mapping it later is a common trap.

Proxy vs non-proxy

  • Lambda proxy: API Gateway passes a standard event; Lambda returns statusCode, headers, and body.
  • Non-proxy/custom: API Gateway owns mapping templates and method/integration response mappings.
  • AWS service integration: VTL can remove a Lambda hop, such as direct API Gateway → DynamoDB.
  • Exam clue: "no code change" plus "transform request/response" usually points to mapping templates.

Status override pattern

{
  "message": "$input.path('$.errorMessage')"
}
#set($context.responseOverride.status = 400)

Use status overrides when the backend response body contains the clue for the client status. Do not choose Lambda proxy if the scenario says API Gateway must perform the mapping.

Exam traps

  • Required query parameter alone is not enough. Add a request validator to enforce it.
  • Method request is client-facing. Integration request is backend-facing.
  • Method response is the declared client contract. Integration response performs the mapping.
  • Proxy integration reduces mapping. Backend code controls the response shape.
  • VTL means REST API mapping templates. It is not Lambda application code.

Practice: place the control

1. A company runs a REST API in Amazon API Gateway in front of a Lambda function that returns customer orders. The public API requires a customerId query string, and invalid requests must be rejected before Lambda is invoked to reduce cost. Which API Gateway feature should the developer use?

Option A is correct because a request validator can enforce required query string parameters before API Gateway calls the backend.
Option B is a distractor because a Lambda authorizer evaluates identity and access, not whether a required query string parameter is present.
Option C is a distractor because usage plans control throttling and quotas for API clients, not request-shape validation.
Option D is a distractor because stage variables pass environment-specific values; they do not reject malformed requests.
Option A is the correct answer. Request validators let API Gateway reject requests with missing required parameters before Lambda runs.
Lesson 0005 covers Lambda configuration. See API Gateway request validators.

2. A travel company exposes GET /reservation?bookingId=... through API Gateway, but the backend is a direct DynamoDB GetItem service integration that expects a JSON payload with TableName and Key. The team does not want to add a Lambda function just to reshape the request. Where should the developer transform the incoming query string into the DynamoDB request payload?

Option A is a distractor because method request settings define the client-facing contract and validation, not the backend payload shape.
Option B is correct because integration request mapping templates reshape client input into the format required by Lambda, HTTP backends, or AWS service integrations.
Option C is a distractor because integration response templates transform backend output after the call returns, not before the backend is invoked.
Option D is a distractor because method responses declare what clients can receive; they do not build backend requests.
Option B is the correct answer. Request-to-backend transformation happens in the integration request, typically with a VTL mapping template for REST APIs.
Lesson 0005 covers API Gateway configuration. See API Gateway data transformations.

3. A software company wants an API Gateway REST API to front a Lambda function. The application code must decide the exact HTTP status code, headers, and response body for each request, and the team wants minimal mapping logic in API Gateway. Which integration type should the developer choose?

Option A is a distractor because mock integrations return static configured responses and do not let Lambda generate dynamic HTTP results.
Option B is correct because Lambda proxy integration lets Lambda return statusCode, headers, and body directly.
Option C is a distractor because non-proxy integrations shift response shaping back into API Gateway mappings instead of letting Lambda own the full HTTP response.
Option D is a distractor because an AWS service integration is for direct calls to services such as DynamoDB or SQS, not for Lambda-controlled HTTP responses.
Option B is the correct answer. Lambda proxy integration is the API Gateway mode where Lambda controls the final status code, headers, and body.
Lesson 0002 covers Lambda invocation types. See Lambda proxy integration.

4. A healthcare application uses an API Gateway REST API with a non-proxy Lambda integration. The Lambda function always returns HTTP 200 with a body such as {"errorMessage":"invalid member id"}, but clients must receive HTTP 400 when that error appears and the backend code cannot be changed. Which API Gateway feature should the developer use?

Option A is a distractor because request validators inspect incoming requests, not backend responses.
Option B is correct because an integration response can inspect the backend payload and use $context.responseOverride.status to change the client status code.
Option C is a distractor because stage variables provide configuration values and do not rewrite status codes.
Option D is a distractor because usage plans manage throttling and quotas, not response mapping behavior.
Option B is the correct answer. In a non-proxy REST API, the integration response is where API Gateway maps backend errors into a different client-facing HTTP status.
Lesson 0005 covers API Gateway configuration. See API Gateway response mapping.

5. A media company is documenting an API Gateway REST API method for video uploads. The method can return HTTP 201 and a custom X-Request-Id header, and that client-facing contract must be declared before response mappings are added. Where should the developer define this contract?

Option A is correct because method responses declare the status codes, headers, and models that the REST API is allowed to return to clients.
Option B is a distractor because integration requests shape what the backend receives; they do not define the public response contract.
Option C is a distractor because Lambda aliases route traffic to function versions and are unrelated to API Gateway response definitions.
Option D is a distractor because authorizer caching stores auth results, not method response metadata.
Option A is the correct answer. Method responses are the client-facing contract for which status codes and headers API Gateway can return.
Lesson 0005 covers API Gateway configuration. See API Gateway method response.
