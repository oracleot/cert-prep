Lesson 0002

Lambda: know how the event enters, then how failure is handled.

DVA-C02 Lambda questions usually hide the answer in the invocation path: synchronous, asynchronous, or event source mapping.

The exam move

Before picking a Lambda feature, ask: who invokes the function? The caller determines retry behavior, error handling, and whether Lambda reads from a source in batches.

Keep the Lambda quick sheet open while you practice.

Three invocation patterns

1. Synchronous

The caller waits. If Lambda succeeds, the caller gets the response. If it fails, the caller sees the error and must decide whether to retry. The AWS CLI command is aws lambda invoke.

2. Asynchronous

The caller hands off the event and does not wait. Lambda queues the event, runs the function, retries function errors twice, and can send invocation records to destinations or a dead-letter queue.

3. Event source mapping

Lambda polls SQS, Kinesis, or DynamoDB Streams and sends records to your function in batches. With Kinesis and DynamoDB Streams, shards are the concurrency unit because each shard is processed in order.

Failure routing mental model

Caller waits? → synchronous → caller handles retry Caller drops event? → asynchronous → Lambda retries, destinations or DLQ Lambda polls source? → event source mapping → source rules, batch behavior, redrive policy

Common DVA-C02 traps

  • Retries require idempotency. Your function may process the same event more than once.
  • Event source mapping is a Lambda resource. Use it for SQS, Kinesis, and DynamoDB Streams.
  • Memory affects CPU. Increasing memory can improve CPU-bound performance.
  • Reserved concurrency is a control. It can cap a function so it does not consume too much account concurrency.
  • Provisioned concurrency targets cold starts. It keeps environments initialized for low latency.

Practice: classify the Lambda path

1. An API Gateway request invokes Lambda and waits for a response before returning to the client. Which invocation type applies?

API Gateway waits for Lambda's response—the caller waits, making this synchronous invocation.
Asynchronous invocation means the caller hands off and does not wait for a response.
Event source mapping means Lambda polls a queue or stream—not direct caller invocation.
Destinations route async failure records—they are not an invocation type.
Synchronous invocation is correct. API Gateway Lambda proxy integration is a synchronous call—the caller (API Gateway) waits for the response.
Review Lesson 0001 for domain triage. Lambda synchronous invocation.

2. S3 sends object-created events to Lambda without waiting for function output. How does Lambda receive these events?

Synchronous means the caller waits—S3 does not wait for Lambda to finish.
S3 invokes Lambda asynchronously and does not wait for output. Lambda retries failed events twice automatically.
Event source mapping means Lambda polls SQS, Kinesis, or DynamoDB Streams—not S3 event notifications.
Provisioned concurrency pre-warms environments—it does not change how S3 delivers events.
Asynchronous invocation is correct. S3 event notifications invoke Lambda asynchronously, and Lambda handles retries.
Lesson 0001 covers Lambda invocation patterns. See Lambda asynchronous invocation.

3. A Lambda function reads records from an SQS queue in batches. The function is triggered automatically when messages arrive. What mechanism delivers these batches?

Synchronous invocation requires a caller that waits for response—Lambda polling a queue is not synchronous.
Asynchronous invocation is triggered by a service calling Lambda—not by Lambda polling a source.
Lambda event source mapping polls SQS queues and delivers batches to the function.
Lambda layers package shared code—they do not deliver queue messages.
Event source mapping is correct. Lambda creates the polling mechanism and manages batch delivery from SQS.
See Lesson 0004. Lambda event source mapping.

4. A production API experiences unpredictable cold-start latency and must respond faster. Which Lambda feature eliminates cold starts for a target concurrency?

Reserved concurrency caps maximum instances—it does not pre-warm environments.
Provisioned concurrency keeps execution environments initialized, eliminating cold starts for targeted function versions or aliases.
Event source mapping handles batch delivery, not latency reduction.
DLQ captures failed events—it has no effect on cold starts.
Provisioned concurrency is correct. It targets specific versions or aliases to maintain pre-warmed instances.
Review Lesson 0003. Lambda concurrency documentation.

5. A function should not consume all available account concurrency, leaving capacity for other functions. Which control prevents one function from monopolizing Lambda?

Reserved concurrency sets a ceiling on a function's concurrent executions, protecting account capacity for other functions.
Provisioned concurrency adds pre-warmed instances—it does not cap concurrency.
Destinations route invocation records on success or failure—they do not control concurrency.
Extensions run alongside the function for monitoring or secrets retrieval—they do not limit concurrency.
Reserved concurrency is correct. It ensures other functions can access account concurrency even if one function scales heavily.
Lesson 0003. Reserved concurrency.

6. Failed asynchronous Lambda events need to be captured for later debugging and reprocessing. Which feature isolates poison-pill events?

A DLQ (SQS or SNS) captures events that exceed Lambda's retry limits during async invocation.
Aliases point to specific versions—they do not capture failed events.
Memory settings affect execution resources—they do not handle failure routing.
Stage variables pass per-environment values to integrations—they do not handle failures.
Dead-letter queue is correct. Lambda sends events to the DLQ after exhausting retries during async invocation.
