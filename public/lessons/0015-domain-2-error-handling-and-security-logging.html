Lesson 0015

Handle failures without leaking security context.

DVA-C02 Domain 2 scenarios test whether you can classify failures, preserve evidence, and avoid exposing secrets while fixing the error path.

The win

When a question says error handling, do not only think retry. Ask whether the app must recover, isolate, redact, or prove who was denied.

Related review: retries, idempotency, and DLQs.

The exam decision path

Transient AWS API failure? → SDK retry mode, exponential backoff, jitter Poison async event or repeated message failure? → Lambda destination / DLQ / SQS redrive policy Multi-step workflow needs alternate branch? → Step Functions Retry and Catch AccessDenied or KMS Decrypt denied? → check explicit deny, SCP, boundary, session, resource policy, key policy Security incident or audit trail needed? → CloudTrail for API calls, CloudWatch Logs for app logs, X-Ray for traces

1. Return safe errors; log useful facts

Client-facing errors should be useful but not revealing. Do not return stack traces, policy names, secret names, access keys, tokens, or raw exception dumps. Log request IDs, principal context, resource IDs, and sanitized failure codes for investigation.

Source: AWS Lambda logging best practices.

2. Pick the failure mechanism by shape

  • SDK call failed temporarily: retry with backoff and jitter.
  • Lambda async event failed: use destinations or a DLQ.
  • SQS message keeps failing: configure a redrive policy.
  • Workflow task fails: use Step Functions Retry and Catch.

Sources: SDK retries, Lambda async errors, and Step Functions error handling.

3. Troubleshoot denial in evaluation order

For IAM, an explicit deny wins. If there is no explicit deny, the request still needs an allow from the right policy path. Check identity policies, resource policies, permission boundaries, session policies, SCPs, and service-specific policies.

KMS has an extra trap: the caller usually needs permission through IAM and the KMS key policy or grant.

Source: IAM policy evaluation logic.

4. Use logs and traces without creating evidence leaks

Use CloudWatch Logs for application security events and CloudTrail for AWS API activity. Use X-Ray to trace requests across services, but keep secrets and sensitive data out of annotations, metadata, and log fields.

Sources: CloudWatch Logs, CloudTrail, and X-Ray concepts.

Exam traps

  • Retry is not authorization. AccessDenied is usually policy evaluation, not transient failure.
  • Logging is not dumping. Never log credentials, bearer tokens, API keys, or decrypted secrets.
  • CloudWatch is not CloudTrail. App logs go to CloudWatch; AWS API history goes to CloudTrail.
  • KMS deny may be key policy. Identity allow alone can still fail.

Practice: classify the safe response

1. An API Gateway Lambda returns raw stack traces and IAM policy names to users. The security team flags this as a data leak. Which response is correct?

Exposing stack traces and policy names leaks internal architecture and attack surface to users.
A generic 500 tells users something failed but does not log evidence for investigation.
Redact client-facing errors while logging request IDs, principal context, and sanitized codes server-side.
Disabling all errors prevents debugging and blocks legitimate troubleshooting.
Redact client-facing errors while logging structured facts server-side. Stack traces, policy names, and credentials must never reach users.
Review safe error patterns and logging and Lambda logging best practices.

2. A Lambda async invocation fails repeatedly after all retry attempts. The event cannot be discarded. Where should failed async events be routed?

Lambda Destinations route async failures to SQS, SNS, Lambda, or EventBridge without data loss.
S3 lifecycle rules handle object expiration—they do not receive Lambda async failures.
CloudFront cache purge clears cached responses—it does not handle Lambda event failures.
DynamoDB TTL expiration removes old items—it does not receive Lambda failure events.
Lambda Destinations or a DLQ routes failed async events to SQS, SNS, or EventBridge for later processing.
Study Lambda async failure destinations and Lambda async invocation errors.

3. A Step Functions workflow needs to branch to a compensation handler when a Lambda task throws a known error. Which feature handles this?

Step Functions Retry retries failed tasks and Catch redirects to an alternate branch when errors are unrecoverable.
EventBridge scheduled rules trigger on time—they do not handle Step Functions task failures.
SQS visibility timeout hides messages during processing—it does not redirect failed workflow tasks.
CloudWatch alarms fire on metric thresholds—they do not redirect Step Functions execution branches.
Step Functions Retry and Catch blocks handle known errors by retrying or routing to a compensation branch.
Study workflow error handling and Step Functions error handling.

4. An IAM role has an explicit allow but requests are still denied. The Lambda function cannot access DynamoDB. What is the next step in troubleshooting?

Increasing Lambda memory does not affect IAM permission evaluation.
CloudWatch metric filters do not reveal IAM policy evaluation results.
Explicit deny in SCPs, permission boundaries, or session policies overrides any allow, regardless of the role's identity policy.
Restarting Lambda does not change IAM policy evaluation logic.
Explicit deny wins in IAM policy evaluation. Check SCPs, permission boundaries, and session policies when an allow is present but access is denied.
Review IAM policy evaluation order and IAM policy evaluation logic.

5. Compliance requires knowing which principal called kms:Decrypt on a customer managed key. Which AWS service records this API call?

CloudTrail records all AWS API calls including kms:Decrypt, capturing principal identity, timestamp, and source IP.
CloudWatch Metrics measure performance—they do not record API call history.
X-Ray traces application request paths—it does not log KMS API calls.
AWS Config records resource configuration changes—it does not log KMS API invocation details.
CloudTrail is correct. It records principal identity, timestamp, and source IP for every kms:Decrypt call.
Study CloudTrail audit trails and CloudTrail user guide.

6. A Lambda function is invoked asynchronously and fails after exhausting all automatic retries. The events cannot be discarded and need to be routed for downstream processing. Which AWS destinations are valid receivers for these failed async events? Choose TWO.

Lambda destinations can route async success and failure events to SQS queues so other consumers can pick up the events for inspection or replay.
CloudFront is a CDN service, not a Lambda destination type. Lambda destinations accept only SQS, SNS, Lambda, or EventBridge as targets.
Lambda destinations can route async success and failure events to SNS topics for fan-out to multiple subscribers.
Step Functions is not a direct destination type. To trigger a state machine from a failed Lambda, route the failure through EventBridge first.
SQS and SNS are valid Lambda destinations for failed async events. Destinations support exactly four target types: SQS, SNS, Lambda, and EventBridge.
Review async failure routing and Lambda async invocation.

7. A distributed application must trace requests across services and query structured application logs for failed requests. Which AWS services are appropriate for this observability need? Choose TWO.

X-Ray trace maps visualize request paths across distributed services and let you filter traces by path, user, or annotation to isolate issues.
CloudWatch Alarms notify operators when metric thresholds breach; they do not record the IAM principal that triggered each alarm action.
Macie discovers sensitive data in S3 buckets—it does not scan CloudWatch Logs streams and does not redact findings from log groups.
CloudWatch Logs Insights uses a structured query language to search and aggregate log entries from Lambda, ECS, API Gateway, and other AWS sources.
X-Ray traces distributed requests and CloudWatch Logs Insights queries structured logs. Macie scans S3, and Alarms notify on thresholds; neither provides the requested observability.
Review tracing and logging patterns and CloudWatch Logs Insights.

8. A team needs durable failure handling for failed messages and orchestrated multi-step workflows in a serverless application. Which statements about durable serverless patterns are valid? Choose TWO.

Dead-letter queues capture failed messages for later inspection or replay, allowing operators to debug poison messages without losing data.
Lambda function DLQ applies only to async invocations; event source mappings from queues use the queue's redrive policy, not a function-attached DLQ.
Step Functions provides durable state machines that replace complex Lambda chaining with built-in retry, catch, and execution history.
Synchronous invocation returns errors directly to the caller with no automatic retry; the caller is responsible for retry logic.
DLQs capture failed messages for replay, and Step Functions provides durable orchestration that replaces fragile Lambda chaining. Function DLQs do not apply to event source mappings, and synchronous invocations do not auto-retry.
Review DLQs and Step Functions patterns and Step Functions error handling.
