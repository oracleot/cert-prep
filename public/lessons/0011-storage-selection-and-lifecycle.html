Lesson 0011

Choose storage by access pattern, not by habit.

DVA-C02 Domain 1 tests whether you can separate object, file, block, relational, key-value, ephemeral, and lifecycle clues quickly.

The win

When a scenario asks for a data store, first classify the data shape: object, shared file, single-instance block, managed database, or temporary scratch.

Related: Domain triage and event service selection.

The exam map

S3 object storage: static assets, backups, data lakes, lifecycle tiers EFS shared Linux file system: many EC2/Lambda/ECS clients, NFS EBS block volume: one EC2 instance, boot/data disk, snapshots FSx managed file systems: Windows, Lustre, NetApp ONTAP, OpenZFS RDS/Aurora relational database: SQL, joins, transactions, managed engines DynamoDB key-value/document: low-latency, serverless, partition-key access /tmp ephemeral scratch: Lambda or EC2 instance-local temporary data

Step 1: Is it storage or a database?

  • S3/EFS/EBS/FSx store files, objects, or blocks. Choose them when the app already knows how to read/write those forms.
  • RDS/Aurora stores relational records when the scenario says SQL, schema, joins, transactions, or existing MySQL/PostgreSQL/Oracle/SQL Server apps.
  • DynamoDB stores key-value/document items when the scenario says serverless scale, single-digit millisecond latency, partition key, or massive throughput.

Step 2: Match the access method

  • Object: S3. You retrieve whole objects by key; great for static content, logs, artifacts, and backups.
  • Shared POSIX files: EFS. Multiple compute clients mount the same elastic Linux file system.
  • Block disk: EBS. One EC2 instance needs a low-latency disk volume or boot volume.
  • Specialized managed file system: FSx. Windows file shares, Lustre HPC, ONTAP, or OpenZFS clues point here.

Step 3: Persistent or ephemeral?

Persistent means the data must survive restarts, instance replacement, or function completion. Think S3, EFS, EBS snapshots/volumes, FSx, RDS/Aurora, and DynamoDB.

Ephemeral means scratch space. Lambda /tmp and EC2 instance store can disappear; use them for caches, decompression, image processing, ETL staging, and temporary downloads.

Lifecycle clues

  • S3 Lifecycle: transition objects to cheaper storage classes or expire/delete objects by age, prefix, tag, or version status.
  • EFS Lifecycle Management: move cold files to lower-cost EFS storage classes based on last access.
  • Amazon Data Lifecycle Manager: automate EBS snapshot and EBS-backed AMI creation, retention, copy, and deletion.
  • Backups are not lifecycle policies. If the clue is database backup/point-in-time recovery, think RDS/Aurora/DynamoDB backup features first.

Fast distractor eliminations

  • If many instances must mount the same Linux directory, eliminate EBS; choose EFS.
  • If the answer needs SQL joins or transactions, eliminate S3 and DynamoDB; choose RDS/Aurora.
  • If the clue is static website hosting or object expiration, eliminate EFS/EBS; choose S3.
  • If the clue is Windows SMB file shares, eliminate EFS; choose FSx for Windows File Server.
  • If the clue is temporary Lambda scratch, eliminate persistent stores unless results must survive.

Practice: choose the store

1. A serverless app stores user uploads and expires raw files after 30 days.

2. Several Lambda functions need the same mounted Linux file directory.

3. A legacy app needs SQL transactions and foreign-key relationships.

4. You must automate daily EBS snapshots and delete old ones.

5. A Lambda function downloads an image, resizes it, then uploads the result.
