Lesson 0007

API Gateway mapping is where requests and responses change shape.

DVA-C02 tests whether you can place validation, parameter mapping, templates, and status-code handling on the correct side of API Gateway.

The win

When a scenario says "transform," "rename," "require," or "return a different status," first classify the flow: method request → integration request → backend → integration response → method response.

Anchor this to Domain 1 development: API Gateway is being tested as the app integration layer.

The five boxes

Client → Method request: public API contract, auth, required params, validation → Integration request: transform client input for Lambda, HTTP, or AWS service → Backend → Integration response: map backend output, headers, errors, status overrides → Method response: public response contract: status codes, models, headers

Method request: reject bad input early

Use method request settings to declare request parameters, such as a required query string. Add a request validator when API Gateway must check required parameters or request bodies before calling the backend.

Integration request: reshape the call

Use integration request mapping when the backend expects a different shape than the public API. In REST APIs, non-proxy integrations can use Velocity Template Language (VTL) mapping templates to build a Lambda event or an AWS service request.

Integration response: reshape the answer

Use integration responses to transform backend output into the public API response. This is where error patterns, headers, body templates, and $context.responseOverride.status can change what the client receives.

Method response: declare the contract

Method responses define which status codes, response headers, and models the API can return. If a header or status is not declared for a REST API method response, mapping it later is a common trap.

Proxy vs non-proxy

  • Lambda proxy: API Gateway passes a standard event; Lambda returns statusCode, headers, and body.
  • Non-proxy/custom: API Gateway owns mapping templates and method/integration response mappings.
  • AWS service integration: VTL can remove a Lambda hop, such as direct API Gateway → DynamoDB.
  • Exam clue: "no code change" plus "transform request/response" usually points to mapping templates.

Status override pattern

{
  "message": "$input.path('$.errorMessage')"
}
#set($context.responseOverride.status = 400)

Use status overrides when the backend response body contains the clue for the client status. Do not choose Lambda proxy if the scenario says API Gateway must perform the mapping.

Exam traps

  • Required query parameter alone is not enough. Add a request validator to enforce it.
  • Method request is client-facing. Integration request is backend-facing.
  • Method response is the declared client contract. Integration response performs the mapping.
  • Proxy integration reduces mapping. Backend code controls the response shape.
  • VTL means REST API mapping templates. It is not Lambda application code.

Practice: place the control

1. An API must reject requests missing ?customerId= before Lambda runs. Which API Gateway component enforces this?

Request validators enforce parameter validation in API Gateway before the request reaches the backend.
Lambda layers package shared code—they do not validate API requests.
Stage caching stores responses—it does not validate request parameters.
Usage plans control throttling and quotas—they do not validate request parameters.
Request validator is correct. It checks required parameters and request bodies before Lambda is invoked.
Lesson 0005 covers Lambda configuration. See API Gateway request validators.

2. You need to convert a public query string into a DynamoDB GetItem payload. Where does this transformation occur?

Method response defines the client-facing contract—it does not transform backend payloads.
Access logging records requests—it does not transform payloads.
Integration request mapping transforms client input into the format the backend expects.
Lambda versions are immutable snapshots—they do not transform API requests.
Integration request is correct. Mapping templates in the integration request transform client input for the backend.
Lesson 0005 covers API Gateway configuration. See API Gateway data transformations.

3. Lambda returns the final statusCode, headers, and body itself. Which API Gateway integration type lets Lambda control the response?

Mock integration returns configured responses without calling a backend—it does not process Lambda responses.
Lambda proxy integration passes the full request to Lambda, which returns statusCode, headers, and body directly.
Model schemas validate request/response bodies—they do not control response handling.
Request models define expected request structures—they do not control response handling.
Lambda proxy is correct. Lambda proxy integration lets Lambda return the full HTTP response directly.
Lesson 0002 covers Lambda invocation types. See Lambda proxy integration.

4. Backend Lambda returns {"error":"bad input"}; the client must receive HTTP 400. Which component changes the HTTP status code?

Integration response override with $context.responseOverride.status changes the HTTP status code.
Reserved concurrency controls maximum executions—it does not change HTTP status codes.
Gateway timeout is a timeout setting—it does not change HTTP status codes.
Stage variables pass per-stage values—they do not change HTTP status codes.
Response override is correct. Integration response mapping can override status codes using $context.responseOverride.status.
Lesson 0005 covers API Gateway configuration. See API Gateway response overrides.

5. An API Gateway REST API must declare that clients can receive a 201 with a custom header. Where is this contract defined? Before integrating, a developer wants to know which HTTP status codes and headers the API Gateway can return so client code can handle all valid responses.

Method response declares the status codes, headers, and models the API can return to clients.
Event sources trigger Lambda invocations—they do not define API response contracts.
Function URLs provide direct HTTPS access to Lambda—they do not use method responses.
Authorizer TTL controls caching of authorizer results—it does not define response contracts.
Method response is correct. Method responses declare the public API contract including status codes and headers.
Lesson 0005 covers API Gateway configuration. See API Gateway method response.
