Lesson 0005

Lambda config traps are mostly "which knob owns this?"

DVA-C02 scenarios often hide the answer in one word: stage, runtime, handler, layer, extension, or package.

The win

You will classify Lambda configuration questions by ownership: Lambda function config, API Gateway stage config, deployment package, layer, or extension.

Review with the Lambda quick sheet after the drill.

The ownership map

Function code + dependencies → deployment package Shared libraries / custom runtimes → Lambda layer Runtime process add-on → Lambda extension Language execution environment → runtime Entry point Lambda calls → handler Per-function values → environment variables Per-API-stage values → API Gateway stage variables

Environment variables vs stage variables

Use Lambda environment variables for values attached to a function version, such as log level or table name for that function.

Use API Gateway stage variables when one API deployment stage must pass different values to the same integration, such as dev vs prod table names.

Runtime, handler, package

  • Runtime: the language environment Lambda uses to run code.
  • Handler: the method Lambda calls when the function is invoked.
  • Deployment package: the code and dependencies uploaded as a zip file or container image.

Layers vs extensions

A layer is reusable content mounted into the execution environment: libraries, dependencies, custom runtimes, or config files.

An extension is a companion process that integrates with the Lambda lifecycle, often for monitoring, security, or configuration tools.

Exam traps

  • Stage clue means API Gateway. "Different table per API stage" points to stage variables plus mapping/integration config.
  • Environment variables are not secret storage. For sensitive values, prefer Secrets Manager or Parameter Store.
  • Handler is not runtime. Runtime selects language; handler selects the entry point.
  • Layer is not invocation logic. Layers package shared dependencies; they do not trigger functions.
  • Container images do not use layers. Package dependencies inside the image instead.

Practice: pick the owner

1. The same Lambda serves dev and prod API stages, each with a different DynamoDB table. How do you pass different table names per stage?

API Gateway stage variables pass per-stage values to integrations—the same function serves both stages with different values.
Lambda layers share code packages—they do not pass per-stage configuration values.
Handler setting defines the entry point—it does not control per-stage configuration.
Runtime selects the language—it does not pass environment-specific values.
Stage variables is correct. API Gateway passes stage-specific values to the Lambda integration without code changes.
Lesson 0007. API Gateway stage variables.

2. A Python Lambda needs shared SDK helper code used by five functions. Which feature avoids duplicating code in each deployment package?

Stage variables pass per-stage values to integrations—they do not package shared code.
Lambda layers are mounted into the execution environment and shared across functions without duplicating in each zip.
Async destinations route invocation records—they do not share code.
Reserved concurrency controls maximum executions—it does not share code.
Lambda layer is correct. Layers mount shared libraries into function execution environments.
Lesson 0003. Lambda layers.

3. Lambda must know which function method to call after invocation. Which configuration specifies the entry point?

Runtime selects the language (Python, Node.js, etc.)—it does not specify the entry point.
Handler specifies the module and method Lambda calls when invoked (e.g., index.handler).
Lambda layers package shared code—they do not define the entry point.
Triggers invoke functions—they do not define the handler method.
Handler is correct. The handler configuration (module.method) tells Lambda which function code to execute.
Lesson 0004 covers Lambda configuration. See Lambda handler.

4. A monitoring agent must run beside the function during lifecycle events. Which Lambda feature runs companion processes?

Lambda extensions run alongside the function during the execution lifecycle for monitoring, security, or configuration tools.
Stage variables pass per-stage values to integrations—they do not run companion processes.
Handler specifies the function entry point—it does not run companion processes.
Timeout controls maximum execution duration—it does not run companion processes.
Extension is correct. Extensions integrate with the Lambda runtime lifecycle and run alongside the function.
Lesson 0004 covers Lambda configuration. See Lambda extensions.

5. A Lambda function connects to an RDS database. Which secure storage should the function retrieve the database password from? A developer wants to avoid hardcoding the password and needs automatic credential rotation without application code changes.

Secrets Manager stores credentials securely with optional automatic rotation and retrieval via SDK or extension.
Stage variables pass configuration values—they are not designed for sensitive credentials.
Lambda layer versions package code—they do not store credentials.
Runtime name selects the language—it does not store credentials.
Secrets Manager is correct. It is designed for sensitive values like database passwords with secure retrieval.
