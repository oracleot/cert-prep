Lesson 0016

Choose the secure config store before wiring Lambda.

DVA-C02 Domain 2 secure-configuration questions usually test one fork: secret, parameter, KMS key, environment variable, or certificate.

The win

When the scenario says credentials, connection strings, encryption, rotation, or environment variables, classify the data before choosing the service.

Related review: Lambda configuration traps and security error handling.

The exam decision path

Database password / API key / rotation required? → AWS Secrets Manager Hierarchical app config or simple encrypted string? → SSM Parameter Store SecureString + KMS Per-function non-secret setting? → Lambda environment variable Encrypt data locally or use envelope encryption? → AWS KMS GenerateDataKey TLS certificate for a website or load balancer? → AWS Certificate Manager Direct control of dedicated HSM hardware? → AWS CloudHSM

1. KMS manages keys, not application secrets

AWS KMS creates and controls cryptographic keys. Use it for server-side encryption integrations and envelope encryption. For local encryption, call GenerateDataKey, encrypt with the plaintext data key, erase it from memory, and store the encrypted data key with the ciphertext.

Source: AWS KMS Developer Guide.

2. Secrets Manager vs Parameter Store

  • Secrets Manager: credentials, API keys, RDS passwords, built-in rotation, KMS encryption, resource policies.
  • Parameter Store: hierarchy paths, config values, versioning, labels, plaintext strings, or encrypted SecureString values.
  • Exam shortcut: if rotation is required, choose Secrets Manager.

Sources: Secrets Manager and Parameter Store.

3. Lambda environment variables are not shared secret storage

Lambda encrypts environment variables at rest. A customer managed KMS key can add control, but environment variables are per function and can be visible to principals allowed to view function configuration. For shared sensitive values, retrieve Secrets Manager or Parameter Store values at runtime, often with a Lambda extension or cache.

Source: Lambda environment variable encryption.

4. Configuration pitfalls the exam likes

  • Stage variable: API Gateway stage-specific value passed to an integration.
  • Environment variable: function-specific setting, not cross-function sharing.
  • SecureString: encrypted Parameter Store value using KMS.
  • Secrets extension: retrieves and caches secrets without writing SDK boilerplate.
  • Interface endpoint: private VPC access to Secrets Manager or SSM APIs.

Exam traps

  • KMS is not where you store the password. KMS protects keys used by services that store data.
  • Parameter Store can be plaintext. Look for SecureString when encryption is required.
  • Lambda env vars are per function. They do not share one secret across functions.
  • CloudHSM is a distractor. Choose it only when the scenario requires dedicated HSM control.
  • ACM is for TLS certificates. It is not for database password rotation.

Practice: choose the secure configuration

1. An RDS MySQL instance needs credentials that are encrypted and rotated automatically every 30 days. Which AWS service stores these credentials?

Secrets Manager stores RDS credentials with built-in automatic rotation at a defined schedule.
Parameter Store plaintext does not support automatic rotation—it requires custom Lambda rotation logic.
ACM manages TLS certificates—it does not store database credentials or support rotation.
Lambda environment variables are per function and do not support automatic credential rotation.
Secrets Manager is correct. It stores database credentials with native automatic rotation on a schedule.
Study Secrets Manager rotation and Secrets Manager.

2. Five Lambda functions in the same VPC need the same encrypted database connection string. The value changes rarely. Which storage approach fits?

SecureString in Parameter Store is encrypted with KMS and can be shared across all Lambda functions via SDK or extension.
Lambda environment variables are per function—duplicating the secret across five functions creates rotation problems.
API Gateway stage variables are for integration configuration—not shared encrypted secret storage.
CloudFront cache keys control what is cached at the edge—not secret storage for Lambda.
SSM Parameter Store SecureString is correct. It stores the encrypted value once and all Lambdas retrieve it at runtime.
Review Parameter Store vs Secrets Manager and Parameter Store.

3. A Lambda function needs a DynamoDB table name that does not change per environment. The value is not a secret. Where should this configuration go?

Lambda environment variables are ideal for per-function non-secret settings like table names, with optional KMS encryption.
Secrets Manager is for credentials and secrets—not plain configuration like table names.
ACM handles TLS certificates—it has no role in Lambda configuration storage.
CloudHSM provides dedicated hardware key storage—it is not used for Lambda configuration values.
Lambda environment variable is correct. Non-secret configuration values belong in Lambda environment variables.
Study Lambda environment variable usage and Lambda environment variable encryption.

4. An application encrypts large files locally using AWS KMS before storing them in S3. Which KMS feature minimizes the performance impact?

GenerateDataKey returns a plaintext data key for local encryption and an encrypted copy for storage—KMS only processes the data key, not the full file.
KMS Encrypt processes all data through KMS, which hits API call limits and quota for large objects.
ACM manages TLS certificates—it does not perform data key generation for envelope encryption.
S3 server-side encryption handles storage encryption—it does not address local encryption use cases.
GenerateDataKey is correct. Envelope encryption uses a data key locally so KMS only processes the small key, not the entire file.
Review KMS envelope encryption and KMS envelope encryption.

5. A Lambda function decrypt fails after switching to a customer managed KMS key. The execution role has iam:kms:*. What else must be checked?

KMS key policy controls who can use the key. Identity permission alone is insufficient if the key policy does not allow the Lambda's principal.
DLQ configuration handles failed event delivery—it does not affect KMS decryption permissions.
Lambda memory does not affect KMS decryption—only key policy and IAM permissions matter.
S3 lifecycle rules manage object expiration—they do not control KMS decryption access.
KMS key policy is correct. KMS requires both IAM permission AND key policy allow for the calling principal.
Review KMS key policy troubleshooting and KMS key policies.

6. A workload running in private subnets must call Secrets Manager APIs without internet access. Which VPC component enables private API access?

Interface VPC endpoints create private ENIs in subnets for AWS services via PrivateLink—no internet required.
NAT Gateway requires an internet gateway and public IPs—private subnets cannot route directly through it.
Lambda function URLs expose Lambda publicly—they do not provide private VPC access to Secrets Manager.
S3 gateway endpoint only provides private S3 access—not Secrets Manager API access.
Interface VPC endpoint is correct. AWS PrivateLink creates elastic network interfaces in private subnets for Secrets Manager.
Study VPC PrivateLink endpoints and VPC endpoints.

7. A team must encrypt objects in Amazon S3 with different controls over key management. Some buckets require audit and customer-managed key policy control; other buckets just need transparent encryption with no operational overhead. Which statements about S3 server-side encryption modes are valid? Choose TWO.

SSE-KMS uses per-object KMS keys, every decrypt call is logged in CloudTrail, and key usage can be controlled via the customer-managed CMK key policy.
SSE-S3 uses a shared AWS-managed key, requires no customer configuration, and is the lowest-overhead server-side encryption option.
SSE-C lets customers supply their own key at request time, but AWS does not store those keys—customers must retain them or data becomes inaccessible.
SSE-S3 uses an AWS-managed key whose policy cannot be modified by the customer. Only customer-managed CMKs under SSE-KMS support custom key policies.
SSE-KMS and SSE-S3 are both valid server-side encryption options. SSE-KMS gives audit and control via customer-managed CMKs; SSE-S3 is fully transparent with AWS managing the key.
Review KMS key policies and S3 encryption.
