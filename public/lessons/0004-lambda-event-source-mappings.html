Lesson 0004

Lambda event source mappings mean Lambda polls and batches for you.

DVA-C02 likes to test whether you can distinguish direct invocation from Lambda reading SQS, Kinesis, or DynamoDB Streams.

The win

You already know Lambda invocation and concurrency. Now add this exam move: when the source is a queue or stream, look for an event source mapping.

Keep the Lambda quick sheet open while you drill.

The pattern

SQS / Kinesis / DynamoDB Streams → event source mapping polls records → Lambda receives a batch → failures are handled by source-specific rules

SQS keyword pattern

Choose event source mapping when Lambda must process queue messages in batches. For failed SQS processing, think visibility timeout and redrive policy on the queue.

Kinesis and DynamoDB Streams keyword pattern

Choose event source mapping when Lambda processes stream records. The exam clue is shards: Lambda processes records from each shard in order, so shard count shapes parallelism.

Async destination distractor

Destinations are for asynchronous Lambda invocations. If Lambda is polling SQS, Kinesis, or DynamoDB Streams, start with event source mapping behavior instead.

Exam traps

  • Poll-based is not async invocation. Lambda is reading from the source.
  • SQS failures belong to SQS controls. Visibility timeout and redrive policy are common answer keywords.
  • Stream ordering matters. Kinesis and DynamoDB Streams preserve order per shard.
  • Shards shape concurrency. More active shards can allow more parallel Lambda processing.
  • Idempotency still matters. Retries can cause duplicate processing.

Practice: spot the mapping

1. A Lambda function must read messages from SQS in batches. The function is triggered automatically when messages arrive. How does Lambda receive these messages?

Destinations route invocation records for async invocations—not batch delivery from queues.
Lambda event source mapping polls SQS and delivers batches to the function.
Provisioned concurrency pre-warms environments—it does not deliver queue messages.
Lambda layers package shared code—they do not deliver queue messages.
Event source mapping is correct. Lambda creates the polling mechanism and manages batch delivery from SQS.

2. A failed SQS message should eventually move to a dead-letter queue after exceeding receive attempts. Which SQS feature controls this behavior?

The redrive policy defines maxReceiveCount and sends messages exceeding that limit to the DLQ.
Stage variables pass per-environment values to integrations—they do not control queue behavior.
X-Ray provides distributed tracing—it does not control message routing.
Aliases point to specific versions—they do not control queue routing.
Redrive policy is correct. It controls when messages move to the DLQ based on maxReceiveCount.
Lesson 0009. SQS DLQ.

3. DynamoDB Streams records must be processed in order per partition key. What is the processing unit for ordered stream delivery?

DynamoDB Streams delivers records per shard in order. Each shard processes its partition key range sequentially.
Lambda layers are for shared code packages—not stream ordering.
Aliases point to versions—they do not define stream processing units.
API Gateway stages represent deployed environments—they do not control stream ordering.
Per shard is correct. DynamoDB Streams preserve ordering within each shard's partition key range.
Lesson 0003 covers Lambda concurrency concepts. See DynamoDB Streams.

4. A Kinesis stream has 25 active shards. What is the key concurrency clue for Lambda processing?

Each Kinesis shard can invoke one Lambda function concurrently. Shard count directly limits Lambda parallelism.
Kinesis Data Streams uses shards, not queues. Queue count is an SQS concept.
Lambda layers package shared code—they do not control parallelism.
API Gateway stages represent deployed environments—they do not limit stream processing.
Shard count is correct. Lambda parallelism for Kinesis is limited to the number of active shards.
Lesson 0003. Lambda with Kinesis.

5. S3 invokes Lambda after object creation and does not wait for output. How does Lambda receive this notification?

S3 event notifications invoke Lambda asynchronously—the notification is handed off and Lambda processes it.
Event source mapping means Lambda polls SQS, Kinesis, or DynamoDB Streams—not S3 notifications.
Poll response is not a Lambda invocation type.
Stream mapping applies to DynamoDB Streams, not S3 notifications.
Async invocation is correct. S3 event notifications trigger Lambda asynchronously.
