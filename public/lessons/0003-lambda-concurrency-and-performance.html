Lesson 0003

Lambda performance starts with concurrency, memory, and cold starts.

DVA-C02 expects you to choose the right Lambda knob from a scenario: cap concurrency, reduce latency, or tune memory/CPU.

The win

After the last lesson, you can classify how Lambda is invoked. Now ask the next exam question: what resource pressure is the scenario describing?

Use the Lambda quick sheet as your reference.

Three knobs to separate

Reserved concurrency

Reserved concurrency sets the maximum concurrent executions for one function. It can protect the rest of the account by preventing one function from using all available concurrency.

Provisioned concurrency

Provisioned concurrency prepares execution environments before requests arrive. Choose it when the scenario demands consistently low latency and cold starts are the problem.

Memory setting

Lambda allocates CPU in proportion to memory. If a function is CPU-bound or memory-bound, increasing memory can reduce duration, but you should tune for cost and speed together.

Cold start mental model

First request or changed function config → Lambda creates execution environment → downloads code / image → initializes runtime and code outside handler → invokes handler Next request may reuse the environment → warm start

Exam traps

  • Do not rely on warmers. Scheduled invocations do not guarantee low latency during scaling or across Availability Zones.
  • Provisioned concurrency is the low-latency answer. Use it when cold starts must be minimized.
  • Reserved concurrency is the blast-radius answer. Use it to cap or guarantee concurrency for a function.
  • Memory tuning changes CPU too. More memory can make CPU-bound functions faster.
  • Lambda Power Tuning uses Step Functions. It tests multiple memory settings and compares speed/cost.

Practice: choose the knob

1. A production API has unpredictable cold-start latency and must respond faster. Which feature eliminates cold starts for a target concurrency level? A team notices latency spikes under load and needs a solution that pre-warms Lambda environments before requests arrive.

Reserved concurrency caps the maximum—it does not pre-warm instances or eliminate cold starts.
Provisioned concurrency keeps execution environments initialized, eliminating cold starts for targeted versions/aliases.
Lambda layers share code—they do not affect cold-start timing.
EventBridge rules route events—they do not control Lambda initialization timing.
Provisioned concurrency is correct. It keeps environments initialized so requests hit warm instances immediately.
Lesson 0002. Provisioned concurrency.

2. One function is consuming too much account concurrency and starving other functions. Which control protects the account? During a traffic spike, one function scales aggressively and errors appear in other Lambda functions that share the account concurrency pool.

Reserved concurrency sets a ceiling on this function's concurrent executions, leaving capacity for other functions.
Provisioned concurrency adds pre-warmed instances—it does not cap a function's maximum concurrency.
Destinations route invocation records on success or failure—they do not control concurrency.
Memory increase changes CPU/resources—it does not limit concurrent executions.
Reserved concurrency is correct. It prevents one function from consuming all account concurrency.
Lesson 0001 covers Lambda concurrency concepts. See Reserved concurrency.

3. A Lambda function runs too slowly and you need a simple config change to test. Which adjustment can improve CPU-bound performance? A developer profiling the Lambda function notices CPU utilization is maxed out during the handler execution, suggesting it needs more compute resources.

Lambda allocates CPU proportionally to memory. Increasing memory gives more CPU power, which can reduce duration for CPU-bound functions.
DLQ captures failed events—it does not affect function execution speed.
Aliases point to versions—they do not change CPU allocation.
Macie is a data security service—it is unrelated to Lambda performance.
Increase memory is correct. Lambda CPU scales with memory, so CPU-bound functions can benefit from more memory.
Lesson 0002 covers Lambda configuration. See Lambda memory configuration.

4. You need to compare several memory sizes for cost and duration automatically. Which AWS feature helps visualize these tradeoffs?

X-Ray traces requests—it does not test memory configurations.
Lambda Power Tuning (uses Step Functions) automatically tests multiple memory configurations and reports cost/duration tradeoffs.
CloudHSM provides hardware security modules—it does not tune Lambda.
CodeArtifact stores package dependencies—it does not analyze Lambda performance.
Lambda Power Tuning is correct. It runs your function across memory settings and reports optimal configurations.
Lesson 0002 covers Lambda invocation and configuration. See Lambda Power Tuning.

5. A Kinesis stream has 100 active shards. What limits parallel Lambda processing of this stream?

Each shard supports one concurrent Lambda invocation. More shards enable more parallel processing.
Queue delay affects visibility timeout—not shard-based parallelism.
Lambda layers package shared code—they do not limit parallelism.
Aliases point to versions—they do not control parallelism.
Shard count is correct. Each Kinesis shard can invoke one Lambda function concurrently, limiting parallelism to shard count.
