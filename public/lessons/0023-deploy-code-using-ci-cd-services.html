Lesson 0023

Deploy code by matching the workload to the CI/CD service.

Task Statement 3.4: select the pipeline, deployment configuration, IaC tool, or rollback control the scenario is really testing.

1. CodePipeline is the release workflow

  • CodePipeline orchestrates source, build, test, approval, and deploy stages after a repository or artifact change.
  • Deploy actions can call CloudFormation, CodeDeploy, ECS, Elastic Beanstalk, Lambda, S3, or custom actions.
  • For cross-account deployment, the pipeline assumes IAM roles in target accounts and needs S3 artifact and KMS key permissions.
  • Manual approval is the exam clue for a human gate before production.

2. CodeBuild proves and packages the change

  • buildspec.yml defines install, pre_build, build, and post_build commands.
  • Use artifacts to pass packaged files to later pipeline stages, and reports to publish test results.
  • Use environment variables, Parameter Store, or Secrets Manager for build-time configuration.
  • Use privileged mode when CodeBuild must build Docker images for ECR.

3. CodeDeploy configurations depend on compute

Traffic-shift shortcut

Lambda and ECS blue/green deployments use canary, linear, or all-at-once traffic shifting. EC2/on-premises deployments use in-place or blue/green, commonly with OneAtATime, HalfAtATime, or AllAtOnce.

Lambda all-at-once → all alias traffic moves immediately Lambda canary → small percent, wait, then the rest Lambda linear → equal percent every interval ECS canary/linear → production listener shifts between task sets EC2 in-place → stop, install, validate, start on same instances EC2 blue/green → provision replacement fleet, reroute traffic
  • Lambda and ECS deployments can be stopped or rolled back by failed hooks or CloudWatch alarms.
  • EC2/on-premises uses the CodeDeploy agent and an AppSpec file for lifecycle hooks and file placement.
  • A new EC2 instance matching the deployment group is not updated until a new deployment is created.

4. IaC and managed platform choices

  • CloudFormation StackSets update the same stack across multiple accounts or Regions with least manual effort.
  • CloudFormation cross-account deployment needs an execution role in the target account and permissions for artifacts and encrypted keys.
  • Elastic Beanstalk supports all-at-once, rolling, rolling with additional batch, immutable, traffic splitting, and blue/green with a separate environment.
  • SAM is the serverless choice for local build/test/debug/deploy of Lambda, API Gateway, and DynamoDB stacks.
  • CDK defines CloudFormation stacks in code; cdk synth emits templates and cdk deploy deploys stacks.

5. Rollback triggers and dynamic deployment clues

  • CodeDeploy auto rollback handles deployment failure or CloudWatch alarm breach.
  • CloudFormation rollback triggers monitor CloudWatch alarms during stack create or update and roll back if they enter alarm.
  • Elastic Beanstalk can roll back unhealthy deployments through deployment health checks and immutable replacement behavior.
  • Stage variables pass API Gateway configuration, such as a DynamoDB table name, to one reusable Lambda function.
  • Parameter Store labels and branches are version and release-management clues.

Retrieval drill: pick the deployment control

1. A commit must trigger source, build, approval, and production stages.

2. Unit tests and package commands must run during the build.

3. Lambda shifts equal traffic increments every ten minutes.

4. ECS production traffic should shift from one task set to another.

5. An EC2 deployment needs file placement and lifecycle hooks.

6. One CloudFormation template must update many accounts and Regions.

7. A serverless API needs local build, test, debug, and deploy.

8. A stack update should roll back if a CloudWatch alarm fires.
