Lesson 0009

Retry safely, then isolate poison failures.

DVA-C02 Domain 1 tests whether you can recognize safe retries, exponential backoff with jitter, idempotent logic, and dead-letter queues in distributed apps.

The win

When a scenario says transient failure, think retry. When it says duplicate side effects, think idempotency. When it says repeatedly failing message, think DLQ.

Related review: Lambda event source mappings and event-driven patterns.

The exam decision path

Is the failure temporary? yes → retry with exponential backoff + jitter no → fail fast or route to an error path Can the operation run twice safely? yes → retry normally no → add idempotency key / conditional write / processed-event record Does the same message keep failing? yes → send to DLQ after max receives / failed async attempts

1. AWS SDK retries are already built in

AWS SDKs include retry behavior for many transient errors, throttling responses, and network failures. Exam clue: if an app uses the AWS SDK and needs reliable API calls, prefer SDK retry configuration before custom retry loops.

Source: AWS SDK retry behavior.

2. Backoff waits longer; jitter spreads callers out

Exponential backoff increases delay after each failed attempt. Jitter randomizes that delay so thousands of clients do not retry at the same time and overload the service again.

Keyword trap: "retry immediately" can make an outage worse. "Reduce contention" or "avoid retry storms" points to jitter.

Source: AWS Architecture Blog: Exponential Backoff and Jitter.

3. Idempotency makes duplicate delivery harmless

Distributed systems can deliver duplicate events. Idempotent logic ensures the original request succeeds and later retries have no additional effect.

  • Use a client request token, order ID, or event ID as an idempotency key.
  • Record processed IDs in DynamoDB, often with a conditional write.
  • Design updates so repeating the same request does not double charge, double email, or double ship.

Source: AWS Lambda Powertools idempotency utility.

4. DLQs are for messages you cannot process now

A dead-letter queue captures failed messages after retry limits so normal processing can continue and operators can inspect, replay, or fix bad events.

  • For SQS, configure a redrive policy and maxReceiveCount.
  • For Lambda async invokes, use destinations or a DLQ for failed events.
  • Do not use a DLQ to hide bugs; alarm on DLQ depth.

Sources: SQS DLQs and Lambda async error handling.

Fast classification

  • ThrottlingException, 5xx, timeout: retry with SDK/backoff.
  • Many clients retry together: add jitter.
  • Payment/order/write may happen twice: make operation idempotent first.
  • One message blocks progress repeatedly: route to DLQ after max attempts.
  • Workflow step needs per-state retry/catch: Step Functions retry and catch policies.

Practice: choose the failure tool

1. A Lambda function writes to a DynamoDB table and receives occasional ThrottlingException errors. The application needs to handle these transient failures automatically. How should the AWS SDK handle these errors?

Throttling is a transient error. SDK backoff retry waits progressively longer between attempts.
DLQ captures repeatedly failing messages—it does not handle transient throttling.
Async Retry applies to Lambda async invocations—it does not change how the AWS SDK handles ThrottlingException on synchronous DynamoDB writes.
Page Iterator walks SDK query responses in chunks—it is not a retry strategy for transient throttling errors.
Backoff retry is correct. Throttling responses should trigger exponential backoff and retry.
Review Lesson 0004 on Lambda error handling. AWS SDK retry behavior.

2. After an AWS Availability Zone recovers from an outage, thousands of Lambda functions retry their invocations simultaneously. The synchronized retries risk overloading downstream services again. Which technique prevents this retry storm?

Jitter adds random delay to retry intervals, spreading callers out to avoid synchronized retries.
ElastiCache caches backend responses to reduce load—it does not randomize retry delays to prevent synchronized retries.
Alarms notify operators—they do not prevent synchronized retries.
Provisioned concurrency eliminates cold starts for steady latency—it does not randomize retry intervals to spread callers out.
Add jitter is correct. Randomizing retry delays prevents thousands of clients from hitting the service simultaneously.
Review Lesson 0009 on backoff and jitter. AWS SDK retry and backoff.

3. A payment service processes a customer order but a network timeout causes a retry. The retry must not create a duplicate charge or a second customer order. Which pattern prevents duplicate side effects from retried requests?

Idempotent code ensures the second execution has no additional effect—the order is created once.
Reserved concurrency caps maximum executions—it does not prevent duplicate side effects.
Provisioned concurrency pre-warms environments—it does not prevent duplicate side effects.
Stage variables pass per-stage values—they do not prevent duplicate operations.
Idempotent code is correct. Idempotency keys or conditional writes ensure duplicate requests have no additional effect.
Review Lesson 0009 on idempotency. AWS SDK retry behavior.

4. A Lambda function consuming from an Amazon SQS queue fails to process the same message repeatedly. After the queue's maxReceiveCount threshold is reached, the message must be removed from the processing queue for inspection. Where does this poison message go after maxReceiveCount?

SQS redrive policy sends messages exceeding maxReceiveCount to the dead-letter queue.
DAX is a DynamoDB read cache—it does not handle SQS message failures.
WAF filters web requests—it does not handle SQS message routing.
EFS provides shared file storage—it does not handle SQS message failures.
Use DLQ is correct. SQS redrive policy routes messages exceeding maxReceiveCount to the dead-letter queue.
Review Lesson 0008 on event-driven patterns. Amazon SQS DLQ.

5. An order fulfillment workflow needs different retry intervals and catch handlers per task state. Some steps should retry on throttling errors, while others should catch specific exception types. Which AWS orchestration service implements per-state retry and catch policies?

Step Functions state machine retry and catch blocks define per-state error handling.
API Gateway handles request/response mapping—it does not implement workflow retry policies.
CloudFront CDN caches content—it does not implement workflow retry policies.
Cognito Sync synchronizes user data—it does not implement workflow retry policies.
Step Functions is correct. State machine definitions include per-state Retry and Catch fields.
